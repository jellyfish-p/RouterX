# RouterX 控制台能力契约

本文档定义 RouterX 控制台和管理接口必须承载的产品能力，不定义网页布局、视觉风格、组件样式或交互动画。后续实现者可以用任意前端形态实现，但必须满足这里的状态、动作、权限、证据和安全边界。

控制台的目标不是把所有配置一次性暴露给用户，而是按用户成熟度分层：

- 小白用户先完成初始化、首个通道、首个 API Key、首次调用、日志和额度确认。
- 站长能管理用户、额度、通道、日志、settings 和基础故障处理。
- 技术用户能配置多 key、多 Base URL、`upstreams`、模型重写、通道分组、路由偏好、计费和观测。
- 运营用户能接入充值码、支付 provider、价格策略、审计和报表，但这些能力不能阻塞 P0 开箱路径。

## 1. 全局原则

### 不做网页设计

本文只规定能力，不规定页面结构。实现时可以是 Web 控制台、命令行控制台、桌面客户端或纯 API 客户端，但用户必须能完成同等任务。

不得把以下内容写成本文范围：

- 颜色、字体、间距、图标和视觉主题。
- 具体路由菜单层级、侧边栏布局或卡片样式。
- 营销页、官网首页或宣传文案。
- 与能力无关的装饰性动画。

### 渐进披露

控制台必须让 L0 用户少做选择，让 L2 用户有足够控制权。

| 用户层级 | 默认展示 | 高级展开 |
|----------|----------|----------|
| L0 小白 | 初始化、首个通道、API Key、调用示例、日志和额度 | 不主动展示复杂路由矩阵 |
| L1 站长 | 用户、额度、通道、日志、设置、基础看板 | 通道高级字段按需展开 |
| L2 技术用户 | 多 key、多 Base URL、分组、模型重写、API Key/channel-group scope、计费规则 | provider-specific 参数、诊断快照 |
| L3 运营用户 | 支付、充值码、价格、审计、报表 | 风控、退款、人工修正和导出 |

### 状态先于操作

每个能力区都应先告诉用户“现在是否可用、为什么不可用、下一步做什么”，再提供修改入口。

统一状态：

| 状态 | 含义 | 必须承载的信息 |
|------|------|----------------|
| `ready` | 可以执行主要动作 | 当前配置摘要、最近成功证据 |
| `empty` | 还没有数据 | 下一步创建动作和最小必要字段 |
| `blocked` | 因依赖缺失不能继续 | 阻塞原因、对应 Runbook、管理员动作 |
| `degraded` | 可用但有风险 | 风险来源、降级策略、建议修复 |
| `forbidden` | 当前身份无权 | 所需角色，不泄露敏感资源 |
| `error` | 操作失败 | 稳定错误 code、request_id、可执行处理建议 |

### 证据链

用户做完一个关键动作后，控制台必须展示可验证结果，而不是只显示“成功”。

| 动作 | 成功证据 |
|------|----------|
| 初始化 | 超级管理员存在、默认 settings 写入、`/ready` 进入可解释状态 |
| 添加通道 | 通道启用状态、模型匹配摘要、密钥脱敏状态、测试结果 |
| 创建 API Key | 明文只展示一次、Token 记录存在、剩余可消耗额度口径明确 |
| 首次调用 | 调用日志、usage、`quota_used`、余额变化、通道命中 |
| 修改 settings | 类型校验通过、缓存刷新、审计摘要、是否需要重启 |
| 支付入账 | 订单状态、provider event、幂等结果、额度变动 |

## 2. 权限和身份边界

控制台登录使用 User JWT；模型调用使用 API Key。两者不能混用。

| 凭据 | 可访问 | 不可访问 |
|------|--------|----------|
| 未登录 | `/health`、`/ready`、setup 状态、已开启的注册登录 | 用户资料、API Key、管理数据、模型调用 |
| User JWT | `/v0/user/*`、自己日志、自己账单、自己 API Key | `/v1/*` 模型调用、管理员接口 |
| Admin User JWT | 普通用户管理、通道、全局日志、基础看板 | 超级管理员 settings、创建同级超级管理员 |
| Super Admin User JWT | settings、管理员账号、高风险配置 | 绕过审计、恢复 API Key 明文 |
| API Key | `/v1/*` 模型调用 | `/v0/user/*`、`/v0/admin/*`、修改任何配置 |

控制台必须避免以下误导：

- 不把管理员 JWT 当作模型调用 Key 展示。
- 不把下游 provider API Key 当作 RouterX 用户 API Key 展示。
- 不提供“查看完整 API Key 明文”的入口。
- 不允许普通用户通过用户端 API Key 编辑所有者或累计已用量；Key 剩余可消耗额度和无限额度标记可由所属用户设置，实际消费仍以用户余额为准。

## 3. 开箱控制台路径

P0 控制台必须能支撑以下路径。

### 3.1 初始化

必须承载：

- 当前初始化状态。
- 创建第一个超级管理员。
- 初始化失败原因的可解释摘要。
- 默认 settings 写入状态。
- 初始化后 `/ready` 状态。

不应承载：

- 多租户、OAuth/OIDC、支付、复杂价格规则等 P0 非必要能力。
- 已初始化后再次创建超级管理员的入口。

验收：

- 空库能初始化一次。
- 已初始化实例拒绝重复初始化。
- 初始化成功后能登录并进入管理员能力区。

### 3.2 首个通道

必须承载：

- provider 类型。
- 通道名称。
- 支持模型列表或模型匹配规则。
- `base_url` 或 `base_urls`。
- `api_key`、`api_keys` 或 `upstreams` 的脱敏摘要。
- 通道启用状态。
- 测试通道和拉取模型能力的结果。

默认模式：

- 小白模式只要求单 `base_url` + 单 `api_key`。
- 高级模式才展示多 key、多 Base URL、`upstreams`、模型重写、优先级、权重和分组。

必须提示：

- `upstreams` 中 base URL 和 key 是绑定关系。
- 外层 `api_keys` 和 `base_urls` 是简化能力，不与 `upstreams` 交叉组合。
- 下游密钥不会在响应和日志中明文返回。

### 3.3 首个 API Key

必须承载：

- API Key 名称。
- 剩余可消耗额度或无限额度。
- 过期时间。
- 创建后明文只展示一次。
- 复制后仍能看到脱敏摘要、状态、创建时间、过期时间和剩余额度。

额度口径：

- 有限 API Key 创建时不扣用户余额，只设置该 Key 的剩余可消耗额度。
- 有限 API Key 成功调用时同时扣用户余额、扣 Key 剩余可消耗额度，并累计 Key 已用额度。
- 无限 API Key 成功调用时扣用户额度，并累计 Key 已用额度。
- 普通用户可以自行调整 Key 剩余可消耗额度或把 Key 设置为自身无限额度；实际消费仍以用户当前余额为准。

### 3.4 第一次调用

必须承载：

- OpenAI-compatible 调用示例。
- API Key 鉴权方式。
- 当前非流式主链路和 OpenAI Chat 基础流式边界。
- 成功后的日志和额度变化入口。
- 常见失败 code 的处理入口。

不应承载：

- 强制用户先理解多协议、支付、企业身份、价格表达式和审计。

## 4. 用户能力区

用户能力区只处理当前用户自己的资源。

| 能力 | 必须显示或返回 | 必须允许动作 | 禁止 |
|------|----------------|--------------|------|
| 个人资料 | username、display_name、email、role、status、quota | 修改 display_name、email、密码 | 自行修改 role、status、quota |
| API Key | name、脱敏摘要、状态、剩余预算、无限标记、过期时间、最近使用目标字段 | 创建、改名、禁用、删除、设置过期时间 | 查看完整明文、转让、提升预算 |
| 调用日志 | 时间、model、status、tokens、quota_used、channel 摘要、error_code | 按 Token、模型、状态、时间筛选 | 查看其他用户日志 |
| 账单统计 | 调用次数、tokens、quota_used、时间范围 | 按时间范围查看 | 把 API Key 预算设置当作模型消费 |
| 故障帮助 | 常见 code、request_id、Runbook 链接 | 根据 code 查看下一步 | 暴露内部密钥或其他用户信息 |

用户端错误展示要求：

- `/v0/user/*` 可以使用 RouterX 统一响应。
- `/v1/*` 示例和 SDK 错误必须展示入口协议兼容格式。
- 所有错误都尽量给出稳定 code 或可映射的原因。

## 5. 管理员能力区

管理员能力区面向站长和运维，必须强调权限、审计和安全动作。

### 5.1 看板

必须承载：

- 用户数。
- Token/API Key 数。
- 通道数和可用通道数。
- 今日调用次数。
- 今日额度消耗。
- 最近错误趋势目标字段。
- `/ready` 和依赖状态目标字段。

看板不能成为唯一事实源；所有汇总应能追溯到日志、账单事实或依赖状态。

### 5.2 用户管理

必须承载：

- 用户列表、搜索、角色、状态、额度、分组、创建时间。
- 创建普通用户。
- 修改普通用户资料、状态、额度和分组。
- 删除或禁用普通用户。
- 超级管理员保护规则。

权限边界：

- 普通管理员可以管理普通用户。
- 普通管理员不能创建、禁用、删除或降级管理员和超级管理员。
- 超级管理员不能误操作禁用或降级自己。
- 所有额度调整必须可审计。

### 5.3 管理员账号

必须承载：

- 管理员列表。
- 创建管理员。
- 修改管理员资料和状态。
- 删除管理员。
- 超级管理员专属操作入口。

目标扩展：

- 双人确认或二次验证。
- 最近登录、最近操作、风险提示。
- 管理员操作审计。

### 5.4 通道管理

必须承载：

- 通道列表、provider、名称、模型、状态、优先级、权重、分组、错误计数、响应时间。
- 创建、编辑、启用、禁用、删除。
- 测试通道。
- 拉取模型。
- 密钥脱敏摘要。
- 通道选择相关的解释字段。

高级字段：

| 字段 | 展示要求 |
|------|----------|
| `api_key` | 只能输入或替换，保存后不回显明文 |
| `api_keys` | 显示数量和脱敏摘要，支持 key 选择模式 |
| `base_urls` | 显示列表和当前选择策略 |
| `upstreams` | 显示绑定关系摘要，强调 base URL 和 key 配对 |
| `model_rewrites` | 显示规则来源、匹配前后模型 |
| `channel_group` | 显示分组和访问控制影响 |
| `upstream_options` | 显示 provider-specific 参数，但必须过滤敏感字段 |

故障状态：

- `no_available_channel`：展示候选过滤摘要。
- `upstream_secret_error`：提示检查 `ENCRYPTION_KEY` 和通道密钥，不展示明文。
- 上游 401/403：提示密钥、模型权限、base URL 或 provider 配置。
- 上游 429/5xx：提示限流、余额、上游状态、`relay.retry_on_status` 白名单、重试和熔断策略。

### 5.5 settings 管理

必须承载：

- 按 category 查看 settings。
- 类型、默认值、当前值、敏感级别、生效方式和阶段。
- 批量修改前校验。
- 敏感值脱敏。
- 修改后是否需要重启。
- 缓存刷新结果。
- 管理审计摘要。

权限边界：

- settings 默认只允许超级管理员修改。
- 普通管理员不得读取敏感原文。
- 不允许用 settings 覆盖启动期必须一致的密钥事实。

## 6. 技术进阶能力区

技术用户需要能解释“为什么这次请求走了这个通道、扣了这个额度、返回了这个错误”。

必须承载：

| 能力 | 必须回答的问题 |
|------|----------------|
| 路由诊断 | 请求模型、候选通道、过滤原因、最终通道、API Key/channel-group scope 是否生效 |
| 协议兼容 | 当前入口协议、支持状态、错误外形、stream 支持阶段，状态以 `docs/PROTOCOLS.md` 为准 |
| provider 参数 | 哪些字段透传、哪些字段被过滤、哪些字段进入 `routerx` |
| usage 结算 | usage 来自上游、估算还是最小扣费 |
| 计费倍率 | 基础额度、模型价格、倍率、折扣、快照版本 |
| 限流熔断 | 命中的限流 key、通道错误计数、熔断原因、恢复策略 |
| 观测追踪 | request_id、error_code、route_snapshot、billing_snapshot，字段语义以 `docs/SNAPSHOTS.md` 为准 |

禁止：

- 让 API Key/channel-group scope 绕过后台访问控制。
- 把 provider-specific 敏感参数透传到日志。
- 用 UI 文案掩盖实际协议不支持。

## 7. 运营能力区

运营能力不是 P0 前置条件，但商业级设计必须预留。

| 能力 | 阶段 | 必须承载 |
|------|------|----------|
| 充值码 | P1/P2 | 创建、发放、核销、额度、过期、审计 |
| 支付产品 | P2 | 商品、金额、额度、币种、启用状态 |
| 支付订单 | P2 | 订单号、provider、金额、状态、用户、过期时间 |
| 支付事件 | P2 | event id、签名结果、幂等结果、入账结果 |
| 退款和人工修正 | P2 | 原因、操作者、前后额度、审计 |
| 报表导出 | P2 | 时间范围、聚合口径、脱敏、导出审计 |

支付控制台必须遵守：

- 不信任客户端提交的金额、额度和商品配置。
- 重复 webhook 不重复入账。
- 签名失败、金额不匹配、订单不匹配时拒绝入账。
- 人工补账必须写审计。

## 8. 观测和故障处理入口

控制台必须把错误 code、日志、指标和 Runbook 串起来。

最低能力：

- 用户能在自己的失败日志中看到 error_code、时间、模型和 request_id 目标字段。
- 管理员能按 user、token、channel、model、status、时间筛选全局日志。
- 管理员能看到通道错误数、最近失败原因和基础延迟。
- 管理员能从 `no_available_channel`、`upstream_secret_error`、`insufficient_quota` 等 code 跳到对应 Runbook。
- 高风险动作进入管理审计。

目标能力：

- 结构化日志。
- Prometheus `/metrics`。
- 告警状态。
- request_id 全链路追踪。
- route_snapshot 和 billing_snapshot。
- 日志保留、导出和脱敏策略。

## 9. 空状态和失败状态

商业级开箱体验必须重视空状态。

| 场景 | 空状态应提示 | 首要动作 |
|------|--------------|----------|
| 未初始化 | 需要创建第一个超级管理员 | 初始化 |
| 无通道 | 需要添加首个上游通道 | 创建通道 |
| 无 API Key | 需要创建调用凭据 | 创建 API Key |
| 无日志 | 尚未产生模型调用 | 查看调用示例 |
| 余额不足 | 当前额度不足 | 管理员加额度或充值 |
| 无支付配置 | 支付为可选能力 | 使用管理员额度或配置支付 |

失败状态必须包含：

- 稳定 code 或可映射原因。
- request_id 目标字段。
- 是否已经调用上游。
- 是否扣费。
- 下一步动作。
- 对应 Runbook。

## 10. 当前实现和目标差异

当前代码已经具备：

- setup 状态和初始化。
- 用户注册、登录、User JWT。
- 用户 API Key 创建、列表、编辑、删除。
- 用户日志和账单统计。
- 管理员用户、管理员账号、通道、日志、settings 和基础 dashboard 接口。
- 通道高级字段的基础管理。
- `/health` 和 `/ready`。
- OpenAI-compatible Chat 非流式转发闭环。

目标仍需补齐：

- 更细的控制台状态模型和机器 code。
- request_id、route_snapshot、billing_snapshot 和 settings_version。
- 管理审计日志。
- 支付、充值码、退款和人工修正。
- 多协议能力的统一可见性矩阵，对齐 `docs/PROTOCOLS.md`。
- 高级 API Key 管理、批量操作、最近使用时间和泄露处理入口。
- 指标、告警和生产 readiness 细节。

## 11. 验收要求

P0 验收：

- 新用户能从控制台或等价接口完成初始化、通道、API Key、首次非流式 Chat 调用、日志和额度确认。
- 普通用户不能看到或修改其他用户资源。
- API Key 明文只展示一次。
- 通道密钥和 settings 敏感值脱敏。
- 管理员能通过日志和错误 code 判断首次调用失败原因。
- 控制台不强制支付、OAuth/OIDC、流式、多协议完整矩阵和复杂价格表达式。

P1 验收：

- 技术用户能解释路由选择、模型重写、通道分组、多 key、多 Base URL 和计费倍率。
- 管理员能看到更完整的失败事实、通道健康和限流熔断状态。
- API Key/channel-group scope 的允许、拒绝和最终结果可追踪。

P2 验收：

- 支付、充值码、审计、指标、告警、企业身份和高级 API Key 管理具备可运营入口。
- 高风险操作都有审计和 Runbook。
- 报表和导出不泄露敏感数据。

## 12. 文档同步

新增或改变控制台能力时，必须同步检查：

- `docs/FLOWS.md`：是否改变用户路径。
- `docs/API.md`：是否有对应接口、鉴权和响应。
- `docs/POLICIES.md`：是否改变策略决策、访问控制、路由偏好、限流或快照。
- `docs/SNAPSHOTS.md`：是否改变调用事实快照、脱敏边界或历史解释。
- `docs/PROTOCOLS.md`：是否改变入口协议、APIType、上游厂商、能力等级或字段降级。
- `docs/DATA_MODEL.md`：是否需要字段、索引或审计表。
- `docs/SECURITY.md`：是否新增敏感信息或权限边界。
- `docs/ERRORS.md`：是否新增错误 code。
- `docs/OBSERVABILITY.md`：是否新增日志、指标或审计事实。
- `docs/RUNBOOKS.md`：是否新增故障处理步骤。
- `docs/TESTING.md`：是否新增验收测试。
- `docs/TRACEABILITY.md`：是否新增能力 ID 或验收证据。

控制台实现可以换皮、换框架、换终端形态，但不能丢掉这些能力合同。
