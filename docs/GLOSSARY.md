# RouterX 术语表

本文档是 RouterX 设计文档的术语权威入口。后续实现和文档更新应优先沿用这里的命名，避免同一能力在 API、数据模型、Relay、计费和运维文档中出现多套说法。

## 使用原则

- 产品、接口和用户路径优先使用用户能理解的词；涉及数据库表、缓存键和服务对象时，再补充代码侧名称。
- 模型路由实体统一称为“通道”；“渠道”只用于支付场景或外部系统原文。
- 对外模型调用凭据统一称为“API Key”；`Token` 只用于 `tokens` 表、缓存对象和服务内部上下文。
- `settings` 是运行时配置权威来源；环境变量只承载启动项和密钥类配置。
- `/v1` 响应和错误外形由入口协议决定，不套 RouterX 管理端统一响应。

## 核心产品术语

| 术语 | 英文/代码名 | 推荐含义 | 边界 |
|------|-------------|----------|------|
| RouterX | gateway | 自部署 AI 大模型中转/网关，负责鉴权、路由、协议转换、计费、日志和观测。 | 不是单一上游厂商 SDK，也不是只做反向代理。 |
| 商业级 | production/commercial grade | 稳定、安全、可观测、可计费、可维护、可进阶配置的默认体验。 | 不表达开源版/商业版授权边界。 |
| 开箱路径 | open-box path | 空库初始化到第一次成功模型调用，并能看到日志和额度变化的最短用户路径。 | 不要求先接入支付、企业身份、完整价格表达式或全协议矩阵。 |
| 调用方 | client | 使用 API Key 调用 RouterX `/v1/*` 的应用、SDK 或脚本。 | 不等同于 RouterX 管理后台用户。 |
| 用户 | user | RouterX 账号主体，拥有额度、API Key、日志和账单。 | 通过 User JWT 访问 `/v0/user/*`，不能直接调用 `/v1/*`。 |
| 服务账号主体 | service account principal | API Key 元数据中描述机器身份、工作负载或外部 IAM subject 的非登录主体。 | 不拥有 User JWT，不可登录后台，也不扩大 API Key 权限。 |
| 管理员 | admin | 具备管理用户、通道、日志和配置权限的 RouterX 用户。 | 管理员的 API Key 也不能继承 `/v0/admin/*` 权限。 |
| 超级管理员 | root admin | 首次初始化创建的最高权限管理员，可管理 settings 和管理员账号。 | 初始化后不可通过普通注册绕过权限边界。 |

## 协议和路由术语

| 术语 | 英文/代码名 | 推荐含义 | 边界 |
|------|-------------|----------|------|
| 入口协议 | ingress protocol | 调用方发给 RouterX 的请求/响应协议外形，例如 OpenAI、Anthropic、Gemini。 | 入口协议决定客户端看到的成功响应和错误格式。 |
| API 类型 | APIType | RouterX 内部区分具体接口能力的枚举，例如 Chat、Models、Embeddings。 | 不等同于上游厂商类型。 |
| 上游厂商 | upstream provider | RouterX 最终调用的模型服务提供方，例如 OpenAI、Anthropic、Gemini、xAI、Azure、Qwen、DeepSeek。 | 产品文档优先用“上游厂商”或“上游服务”。 |
| 能力矩阵 | capability matrix | `docs/PROTOCOLS.md` 中记录入口协议、APIType、上游厂商、字段转换、流式、usage、错误和测试状态的矩阵。 | 不能用“路由已注册”替代“能力已兼容”。 |
| Provider | provider | 适配器和配置中的厂商标识，也可用于支付模块的支付提供方。 | 模型 provider 和支付 provider 必须结合上下文区分。 |
| Adapter | adapter | `internal/relay` 中隔离厂商差异的适配器。 | Adapter 不负责用户权限和扣费策略。 |
| 通道 | channel / `channels` | 一个可被路由选择的模型上游配置实体，包含类型、密钥、Base URL、模型、优先级、权重和分组。 | 模型路由文档不要再称为“渠道”。 |
| 通道内部上游 | upstream target / `upstreams` | 同一个通道内绑定的一组上游地址和密钥，通常是 `base_url + api_key` 成对出现。 | 不是独立通道，不能参与跨通道优先级排序。 |
| Base URL | base_url / base_urls | 上游服务根地址。 | 多个 Base URL 只是通道内部目标，不代表多个通道。 |
| 模型名 | model | 调用方请求中的模型名。 | 日志 P0 记录调用方模型名。 |
| 上游模型名 | upstream model | 模型重写后真实发给上游的模型名。 | 需要和调用方模型名分别记录，便于账单和排障。 |
| 模型重写 | model_rewrites | 将调用方模型名改写为上游模型名的通道规则。 | 不应改变调用方看到的入口协议。 |
| 路由偏好 | API Key/channel-group scope | 调用方在请求中给出的通道分组、provider 等偏好。 | 只能收窄后台允许的候选集，不能越权。 |

## 账号和凭据术语

| 术语 | 英文/代码名 | 推荐含义 | 边界 |
|------|-------------|----------|------|
| User JWT | JWT | 登录 RouterX 控制台和 `/v0/user/*`、`/v0/admin/*` 的会话凭据。 | 不能用于 `/v1/*` 模型调用。 |
| API Key | API key / `sk-*` | 调用方用于 `/v1/*` 的模型调用凭据，明文只返回一次。 | 对外文档和用户路径优先使用这个词。 |
| Token | `tokens` | 数据库和服务层保存 API Key 哈希、状态、过期时间和额度的内部实体。 | 不要在用户界面或产品路径中把它解释成模型 token。 |
| 模型 token | usage token | 模型请求和响应中的用量单位，例如 prompt tokens、completion tokens。 | 需要小写或加限定词，避免和 `Token` 实体混淆。 |
| API Key 哈希 | SHA256 hash | 数据库存储的 API Key 哈希值。 | 明文 API Key 不进数据库、不进日志、不进缓存键。 |
| 上游密钥/下游密钥 | upstream secret | 通道中用于调用上游厂商的 API Key 或等价密钥。 | 必须加密存储和脱敏展示，不能由调用方请求覆盖；设计文档优先写“上游密钥”，描述 HTTP 调用方向时可写“下游密钥”。 |

## 额度和计费术语

| 术语 | 英文/代码名 | 推荐含义 | 边界 |
|------|-------------|----------|------|
| 用户额度 | `users.quota` | 用户账户层面的可用余额，单位为基础额度单位。 | 无限 API Key 调用会扣用户额度。 |
| API Key 预算上限 | `tokens.quota_limit` | 有限 API Key 的剩余可消耗额度。 | 不来自用户余额划拨；调用成功时同时扣用户余额、扣 Key 剩余额度并累计 Key 已用量。 |
| 无限 API Key | `quota_limit=-1` / `unlimited=true` | 不限制 API Key 自身额度，但仍受用户额度和系统策略约束。 | 不是免费调用，也不能绕过限流和访问控制。 |
| Key 预算 | key budget | 对单个 API Key 的剩余可消耗限制。 | 创建或调整预算不等于模型消费，也不改变用户余额。 |
| 消费 | consumption | 模型调用成功后产生的 `quota_used`。 | 用户账单按消费日志聚合。 |
| usage | usage | 上游或适配器给出的模型 token 用量。 | usage 不是最终扣费金额。 |
| `quota_used` | quota_used | RouterX 结算后的本次额度消耗。 | 账单聚合以它为准。 |
| 价格表达式 | price_expression | 将 usage 和变量转换为基础费用的规则文本。 | 用户分组倍率、通道分组倍率不应硬塞进表达式。 |
| 倍率快照 | multiplier_snapshot | 本次调用使用的用户分组、通道分组等倍率事实。 | 用于历史账单解释，不能用当前规则重算旧账。 |
| 调用事实快照 | call fact snapshot | 一次调用的请求、策略、路由、usage、计费、错误和 settings 摘要。 | 统一契约见 `docs/SNAPSHOTS.md`，不保存密钥或敏感正文。 |

## 配置和运行术语

| 术语 | 英文/代码名 | 推荐含义 | 边界 |
|------|-------------|----------|------|
| settings | `settings` | 运行时配置权威来源，保存业务开关、默认值、倍率、限流和日志策略。 | 不为业务配置另建并行权威来源。 |
| 环境变量 | env | 启动期配置和不可丢失密钥来源，例如 `SQL_DSN`、`LOG_SQL_DSN`、`REDIS_CONN`、`JWT_SECRET`、`ENCRYPTION_KEY`。 | 不承载计费倍率、通道访问策略等运行时业务配置。 |
| 配置注册表 | settings registry | 服务层维护的 key、类型、默认值、校验、生效方式和敏感级别定义。 | 即使数据库表简单，代码也需要有等价注册表。 |
| `/health` | liveness | 进程存活探测。 | 不做数据库、迁移、密钥等深度检查。 |
| `/ready` | readiness | 服务是否适合接收真实流量的就绪探测。 | 生产模式应检查关键 settings、密钥和依赖策略。 |
| 日志 | logs | 模型调用事实，包含请求身份、路由、usage、扣费和错误摘要。 | 成功消费账单以日志或明确账本事实为准。 |
| 审计日志 | audit logs | 配置、额度、支付、管理员操作等管理行为记录。 | 不替代模型调用日志。 |

## 分组和访问控制术语

| 术语 | 英文/代码名 | 推荐含义 | 边界 |
|------|-------------|----------|------|
| 用户分组 | user_group | 用户所属计费、访问控制或运营分组。 | 新用户默认归入 `default`，不等同于管理员角色。 |
| 通道分组 | channel_group | 通道的路由、访问控制和倍率分组。 | 新通道默认归入 `default`，设计和配置优先使用 `channel_group`。 |
| 模型分组 | model_group | 目标设计中可选的模型聚合维度。 | 若实现阶段只使用通道分组，文档需保持 `channel_group` 为主。 |
| 策略 | policy | RouterX 对身份、额度、限流、模型、通道分组、路由偏好和计费倍率的决策规则集合。 | 不等同于单个前端开关或单个 settings key。 |
| 访问控制 | access control | 判断用户、API Key、通道、模型和分组是否允许本次调用。 | 应发生在上游调用前。 |
| 策略快照 | policy snapshot | 本次路由、访问控制和计费规则的结构化摘要。 | 用于排障和审计，不应包含密钥原文。 |
| 作用域 | scope | API Key 或后续策略对模型、通道分组、API 类型、请求类型、IP 和预算的收窄规则。 | 只能收窄，不能放大用户或系统允许范围。 |

## 容易混淆的命名

| 不推荐混用 | 推荐写法 | 说明 |
|------------|----------|------|
| 模型“渠道” | 模型“通道” | `channels` 表和 Relay 路由实体统一叫通道。 |
| 支付“通道” | 支付“渠道”或支付 provider | 支付语境保留中文自然说法，避免和模型通道混淆。 |
| Token 表示模型用量 | 模型 token 或 usage token | `Token` 在 RouterX 代码中优先指 API Key 内部实体。 |
| 下游厂商 | 上游厂商或上游服务 | 产品设计优先用上游；描述 HTTP 调用动作时可写“下游请求”。 |
| 系统配置表 | settings | 统一使用 `settings` 表达运行时配置。 |
| API Key 明文存储 | API Key 哈希存储 | 明文只在创建时返回一次。 |

## 文档更新要求

- 新增接口、字段、settings key、错误 code 或日志字段时，如果引入新概念，应先补充本术语表。
- 修改计费、Relay、账号、API Key、策略或配置文档时，应检查是否把 API Key、Token、模型 token、通道、上游、provider、策略和作用域混用。
- README 和 `docs/DESIGN.md` 只保留总览表达；具体细节回链到专题文档和本术语表。
