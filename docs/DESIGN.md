# RouterX 商业级设计总稿

## 1. 定位

RouterX 是一个面向自部署场景的商业级 AI 大模型中转和网关项目。它的目标不是只做一个简单反向代理，而是提供一套可以被个人站长、小团队和企业私有部署直接使用的模型访问基础设施。

商业级在本项目中的含义是工程质量和可运营能力，而不是开源版和商业版授权边界。默认设计必须满足：

- 稳定：空库初始化、迁移、鉴权、转发、扣费和日志形成闭环。
- 安全：用户密码哈希、API Key 只返回一次并哈希存储、下游密钥加密存储、日志默认脱敏。
- 可观测：健康检查、就绪检查、调用日志、用量统计、通道状态、后续指标和审计日志。
- 可计费：用户额度、API Key 最大消耗额度、模型价格、倍率、扣费事务和账单统计一致。
- 可维护：文档与代码阶段一致，配置集中，迁移版本化，错误边界明确。
- 可进阶：懂技术的用户可以配置多通道、多 key、模型重写、通道分组、协议扩展和计费策略。

RouterX 不以网页设计为本文档范围。控制台和接口只需要承载必要的产品能力：初始化、账号、API Key、通道、日志、设置、计费和后续支付等流程。

本轮新增的设计意图：

- 新用户默认归入 `default` 用户分组，新通道默认归入 `default` 通道分组；空分组在策略层归一为 `default`。
- API Key 的最大消耗额度是预算上限，不在创建时划拨用户余额；有限 Key 调用时同时受用户余额和 Key 预算约束。
- 直接启动单镜像时使用 SQLite 开箱路径，Redis 可省略；配置 PostgreSQL/MySQL 等外部数据库或集群模式时必须配置 Redis。
- `LOG_SQL_DSN` 可把高流量调用日志写入独立日志库，但扣费所需的最小结算事实必须保留在主库或 outbox。
- 模型到通道候选集应按模型、APIType、用户分组、通道分组等维度预加载和缓存；管理员修改后通过版本号和 Redis 失效广播保证集群一致性。

## 2. 设计哲学与产品取舍

RouterX 的第一原则是开箱稳定，第二层体验是极致可控，运营赚钱能力是可选增强。实现和文档都应围绕这个顺序取舍，不能把高级配置和商业运营复杂度提前压到第一次使用路径里。

### 第一原则：开箱稳定

默认路径必须让用户先完成一次可靠调用，再理解系统的高级能力。第一次使用只需要做五件事：

```text
初始化管理员 -> 添加第一个通道 -> 创建 API Key -> 完成第一次调用 -> 查看日志和额度变化
```

在这条路径里，系统应该提供明确默认值，隐藏非必要选择。用户不需要先理解多协议转换矩阵、计费表达式、熔断策略、OAuth/OIDC 或支付回调，也能把服务跑通。

### 第二层体验：极致可控

当用户进入进阶场景时，RouterX 必须允许深度配置，但每个高级能力都要具备清晰边界：

- 多通道、多 key、模型重写、通道分组必须能解释“为什么这次请求选中了这个通道”。
- `routerx` 扩展参数必须有安全过滤和冲突规则，不能把鉴权 header 或内部字段交给用户任意覆盖。
- 计费表达式、倍率和访问控制必须保存规则快照，历史账单不能因规则变更失去解释。
- 熔断、限流、重试和降级必须有默认策略，也要能被技术用户调优。

### 第三层体验：运营能力可选

额度、价格、扣费、日志和账单一致性是商业级核心能力；支付、充值码、OAuth/OIDC、企业审计和高级风控是可选增强。它们应该能接入完整商业闭环，但不能成为自部署最小可用路径的前置依赖。

### 固定设计决策

这些决策是后续实现和文档维护的默认锚点，除非有明确代码变更和迁移计划，不应在专题文档中反向改写。

| 决策 | 默认取舍 |
|------|----------|
| 产品授权边界 | 文档中的商业级指工程质量和运营能力，不区分开源版/商业版授权边界。 |
| 配置权威来源 | 数据库 `settings` 是运行时配置权威来源，环境变量只承载启动期和密钥类配置。 |
| 开箱路径 | 初始化、首个通道、首个 API Key、首次调用、日志和额度变化是 P0 主线。 |
| 公开注册 | 自部署商业级默认关闭公开自助注册，由管理员创建用户或显式开启。 |
| 路由优先级 | 后台策略、安全过滤和访问控制优先于用户请求中的 `routerx.route` 偏好。 |
| 模型 API 响应 | `/v1` 必须保持入口协议兼容响应和错误格式，不返回 RouterX 管理端统一包装。 |
| 密钥策略 | API Key 明文只返回一次并哈希存储；下游密钥使用 `ENCRYPTION_KEY` 或 KMS 加密。 |
| 计费策略 | 额度、价格、日志、扣费和账单一致性是核心；在线支付是可选运营模块。 |
| 页面设计 | 本文档不设计网页，只约束控制台和接口必须承载的产品能力。 |

### 复杂度归属

复杂度应该优先由系统吸收，而不是转嫁给用户。

| 复杂点 | 系统应承担 | 用户应看到 |
|--------|------------|------------|
| 协议差异 | Adapter 转换、字段降级、错误映射 | 清晰的不支持错误或兼容响应 |
| 通道路由 | 候选过滤、优先级、权重、熔断、重试 | 可解释的通道选择和日志 |
| 计费规则 | usage 提取、表达式执行、倍率快照、事务扣费 | 可追溯账单和余额变化 |
| 密钥安全 | 哈希、加密、脱敏、只返回一次 | 不泄露密钥的管理体验 |
| 运行配置 | settings 默认值、缓存、动态刷新 | 少量必要配置和明确开关 |
| 错误语义 | 协议兼容错误、错误分类、日志摘要 | 知道该修请求、充值、换 key 还是找管理员 |

## 3. 目标用户与体验

RouterX 同时服务两类用户。

| 用户 | 目标体验 |
|------|----------|
| 小白自部署用户 | 能按文档完成初始化、添加第一个通道、创建 API Key、完成第一次 SDK 调用，并能看到日志和额度变化。 |
| 技术用户和企业用户 | 能理解路由、协议、通道、计费、密钥、迁移、缓存、限流和观测规则，并能按需扩展或接入现有系统。 |

小白体验不等于弱设计。默认路径应该隐藏非必要选择，但系统底层仍采用生产可演进的模型。技术用户体验不等于配置堆叠，所有进阶能力必须有清晰边界、默认值和失败模式。

体验分层：

| 层级 | 面向用户 | 能力范围 |
|------|----------|----------|
| L0 开箱路径 | 小白自部署用户 | 初始化、首个通道、首个 API Key、首次调用、日志和余额确认 |
| L1 管理路径 | 站长和小团队管理员 | 用户、Token、通道、日志、设置、基础计费和基础故障处理 |
| L2 进阶路径 | 技术用户和企业用户 | 多协议、多上游、路由偏好、计费策略、限流熔断、观测审计 |
| L3 运营路径 | 平台运营者 | 充值码、支付插件、用户分组、价格策略、风控和报表 |

## 4. 默认开箱路径契约

默认开箱路径是 RouterX 的产品主线。任何新增能力都不能破坏这条路径。

```text
空系统
  -> 初始化超级管理员
  -> 管理员添加第一个可用通道
  -> 用户或管理员创建第一个 API Key
  -> 使用 OpenAI-compatible SDK 或 curl 完成第一次调用
  -> 在日志和额度中看到本次调用结果
```

每一步都要有明确的成功状态和失败解释。

| 步骤 | 成功状态 | 失败解释 |
|------|----------|----------|
| 初始化 | 系统存在一个启用的超级管理员，默认 `settings` 已写入 | 已初始化、密码不合规、数据库错误、JWT 配置不可用 |
| 添加通道 | 通道启用，密钥已加密或安全保存，模型匹配规则明确 | 密钥缺失、URL 不可用、provider 类型不支持、测试失败 |
| 创建 API Key | 返回一次性明文 `sk-`，数据库保存哈希 | 用户禁用、额度不足、过期时间非法 |
| 第一次调用 | `/v1` 返回 SDK 兼容响应，日志记录成功，额度扣减 | API Key 无效、余额不足、无可用通道、下游错误、请求不兼容 |
| 查看结果 | 用户能看到自己的日志和账单，管理员能看到全局日志 | 权限不足、日志被清理、统计延迟或配置禁用 |

默认路径的约束：

- 不能要求用户先配置支付、OAuth/OIDC、多通道、模型价格表达式或高级观测。
- 不能把高级路由参数作为第一次调用的必要条件。
- 不能让下游错误以内部 panic、空响应或 RouterX 管理端响应格式暴露给 SDK。
- 不能在任何步骤泄露 API Key、下游密钥、数据库 DSN 或支付密钥。
- 不能让“看起来成功”的调用缺少日志或扣费事实，除非该请求明确是免费测试或管理员通道测试。
- 不能让第一次验证调用被初始额度为 0 卡死；初始化应为超级管理员写入可配置启动额度，或提供明确的管理员额度调整步骤。

首次调用失败时，系统必须给出能指导下一步的错误语义：

| 失败类型 | 用户应理解 | 管理员应能看到 |
|----------|------------|----------------|
| API Key 无效或过期 | 更换或重新创建 API Key | token id、所属用户、禁用/过期/删除原因 |
| 余额不足或限流 | 充值、降低并发或联系管理员调整额度 | 用户额度、Key 预算、限流命中项 |
| 请求不兼容 | 修改 model、body、stream 或不支持字段 | 入口协议、API 类型、解析错误 code |
| 无可用通道 | 当前模型没有可用上游 | 候选通道过滤原因、模型匹配和熔断状态 |
| 通道密钥或配置错误 | 联系管理员检查上游配置 | 通道 ID、provider、密钥解密状态和下游 401/403 摘要 |
| 下游临时故障 | 稍后重试或换模型 | 下游状态码、超时、重试次数和通道错误计数 |

默认路径完成后，系统应自然引导用户进入进阶能力：多通道、流式响应、更多协议、计费规则、限流熔断、支付插件和观测审计。

### 控制台承载能力

本文档不设计网页布局、视觉风格或页面组件，只规定控制台和接口必须承载的任务。

| 能力区 | 必须完成的任务 | 必须看见的结果 |
|--------|----------------|----------------|
| 初始化 | 判断是否已初始化，创建第一个超级管理员，写入默认 `settings` | 初始化状态、管理员创建结果、JWT 配置可用性 |
| 账号 | 登录、查看自己资料、修改资料和密码，管理员创建和管理普通用户 | 当前身份、角色、状态、额度和禁用原因 |
| API Key | 创建、查看、编辑、禁用、删除 Key，创建时只展示一次明文 | Key 名称、状态、过期时间、剩余额度、最近使用状态 |
| 通道 | 创建第一个通道，测试连通性，查看模型，启用/禁用通道 | 通道状态、支持模型、优先级、权重、最近错误和密钥脱敏状态 |
| 调用验证 | 用 OpenAI-compatible `/v1/models` 和 `/v1/chat/completions` 验证闭环 | SDK 兼容响应、错误原因、日志和额度变化 |
| 日志账单 | 查询自己的调用记录，管理员查询全局日志和统计 | user/token/channel/model/usage/quota/status/error 摘要 |
| 设置 | 管理运行时 settings，敏感项脱敏，变更可审计 | 配置分类、当前值、默认值、是否需要重启或刷新缓存 |

控制台不应把高级能力变成开箱门槛。第一次调用前不应强制配置支付、OAuth/OIDC、复杂计费表达式、多通道策略、完整观测系统或企业审计。

## 5. 技术用户进阶能力地图

技术用户的进阶体验应该按“我要控制什么”来组织，而不是按代码目录组织。

| 能力组 | 用户意图 | 主要设计点 |
|--------|----------|------------|
| 路由控制 | 控制请求走哪个上游、如何容灾、如何区分套餐 | 通道优先级、权重、多 base URL、多 key、模型重写、通道分组、`routerx.route` 偏好、熔断和重试 |
| 协议控制 | 兼容不同 SDK 和上游厂商 | OpenAI/Anthropic/Gemini 入口协议、上游 provider adapter、字段降级、错误映射 |
| 计费控制 | 控制价格、折扣、访问权限和账单解释 | 模型价格表、通道价格覆盖、用户分组倍率、通道分组倍率、规则快照 |
| 可靠性控制 | 控制限流、超时、重试、失败隔离 | Redis 限流、请求超时、响应大小限制、错误计数、自动熔断、半开恢复 |
| 观测审计 | 解释系统行为、排查问题、满足审计 | 调用日志、管理审计、结构化日志、指标、request_id、规则快照 |

进阶能力必须遵守三个约束：

- 可组合：通道分组、用户分组、计费倍率和路由偏好可以组合，但优先级必须固定。
- 可解释：每次请求最终选中的通道、计费规则、失败原因和限流状态应能从日志或审计中还原。
- 可降级：SQLite 单机模式可无 Redis；外部数据库或集群模式缺 Redis 必须明确失败。某个通道、某个 provider adapter 或某类高级配置不可用时，系统应有明确失败或降级策略。

## 6. 当前代码事实

文档必须以当前代码为基础，而不是旧路线图。当前仓库已经具备以下基础能力：

- 后端框架、路由分组、数据库初始化和嵌入式 SQL 迁移。
- SQLite、PostgreSQL、MySQL 多数据库支持。
- 首次初始化超级管理员和默认 `settings`。
- 用户注册、统一登录、User JWT 鉴权和管理员角色校验。
- 用户、管理员、API Key、通道、日志、设置等基础接口。
- API Key 使用 `sk-` 明文创建，明文只在创建响应出现，数据库保存 SHA256 哈希，并兼容早期明文存量迁移。
- 下游通道 API Key 支持加密存储，通道支持单 base URL、多 base URL、单 key、多 key、key 选择模式、模型重写、通道分组和扩展配置。
- `/v1` 已注册 OpenAI、Gemini、Anthropic 相关入口路由，已有 OpenAI-compatible 非流式转发和模型列表基础能力。
- 日志查询、用户账单统计和管理端看板已有基础接口。
- `/health` 和 `/ready` 已存在。

仍需要继续补齐的商业级能力：

- 流式响应完整闭环和流式 usage 结算。
- 多协议入口和多上游转换矩阵的完整语义映射。
- 更完整的模型价格表、通道价格覆盖、倍率和访问控制。
- 充值码、在线支付、支付回调幂等和支付审计。
- 结构化日志、Prometheus 指标、管理审计日志、告警和生产安全基线。
- OAuth/OIDC、验证码、账号恢复、注销保留账号等企业账号增强。

## 7. 配置原则

配置权威来源是数据库 `settings` 表。环境变量只承载启动期和不可丢失的密钥类配置。

| 配置来源 | 用途 | 示例 |
|----------|------|------|
| 环境变量 | 启动必须项、跨实例必须一致的密钥、外部服务密钥 | `SQL_DSN`、`LOG_SQL_DSN`、`REDIS_CONN`、`JWT_SECRET`、`ENCRYPTION_KEY`、支付密钥 |
| `settings` | 运行时配置、功能开关、默认值、计费和限流策略 | `jwt.user_expire_hours`、`relay.timeout`、`billing.default_ratio` |
| SQL 表 | 需要强查询和版本审计的业务实体 | 用户、API Key、通道、日志、模型价格、订单 |

settings 的生命周期必须清晰：

| 阶段 | 设计要求 |
|------|----------|
| 定义 | 每个 key 都要有分类、默认值、类型、校验规则、是否敏感、是否需要重启或刷新缓存。 |
| 初始化 | 首次初始化幂等写入默认值，不覆盖管理员已修改的值。 |
| 读取 | 业务代码通过 typed accessor 读取，不在各处散落字符串解析。 |
| 修改 | 管理端修改先校验，再写 DB，刷新缓存，并记录审计摘要。 |
| 生效 | 可热更新项即时生效；需要重启的项必须明确提示，不能静默半生效。 |
| 回滚 | 配置错误应能根据审计记录恢复到上一个有效值。 |

设计约束：

- `JWT_SECRET` 在生产和多实例中必须固定一致；未配置时只允许初始化流程生成并写入 `settings`。
- `ENCRYPTION_KEY` 不能随机变化，否则已加密的下游密钥无法解密。
- 运行时可调配置统一通过 `settings` 读取，后续可用 Redis 缓存。
- 文档中不再引入另一套独立权威配置表；如未来需要 JSON 配置能力，应作为 `settings.value` 的 JSON 内容表达。

开箱路径可以允许单镜像 SQLite 模式用最少配置启动，但生产就绪必须暴露关键风险：缺少固定 JWT 密钥、缺少加密主密钥、迁移异常、外部数据库模式缺少 Redis、Redis 策略不符合生产要求或关键 settings 非法时，`/ready` 应返回不就绪，而不是让实例继续承接真实流量。

## 8. 核心模块设计

### 初始化

空数据库启动后，系统必须进入未初始化状态。初始化流程创建第一个超级管理员、写入本地用户名密码身份、写入默认 `settings`，并具备并发保护。

初始化应写入开箱所需的最小默认值，包括 JWT、限流、Relay、日志和启动额度。启动额度建议使用 `billing.bootstrap_admin_quota` 表达，默认只服务于首次验证和管理员自测；正式运营额度仍由管理员通过用户额度和计费规则管理。

验收标准：

- 未初始化时 `/v0/setup/status` 返回 `initialized=false`。
- 首次 `POST /v0/setup/init` 成功后系统进入已初始化状态。
- 重复初始化返回冲突错误。
- 未初始化时非 setup/health 请求被拦截，`/v1` 返回协议兼容错误而不是 RouterX 管理端响应包装。
- 初始化后的超级管理员具备足够完成首次验证调用的额度，或系统明确提示需要先调整额度。

### 账号与权限

用户资料和登录身份分离。所有账户必须拥有用户名和密码，本地密码保存在 `user_identities(username, local)` 的 `password_hash` 中。

默认账号策略：

- 用户名密码登录强制可用，不能被关闭。
- 自部署商业级默认关闭公开自助注册，由管理员创建用户或后续开启注册。
- 一旦开启自助注册，必须配套验证码、限流和身份去重。
- OAuth/OIDC 不能创建无密码账户，首次注册必须补齐用户名和密码。

权限层级：

| 角色 | 能力 |
|------|------|
| 普通用户 | 管理自己的资料、API Key、日志和账单。 |
| 管理员 | 查看和管理普通用户、通道、日志和看板。 |
| 超级管理员 | 管理管理员账号和系统设置。 |

管理员不使用独立登录入口，统一通过 `/v0/user/login` 获取 User JWT，再由 `/v0/admin/*` 校验角色。

权限边界必须按“凭据类型”和“角色”同时判断：

| 凭据/角色 | 允许 | 禁止 |
|-----------|------|------|
| 未登录请求 | `/health`、`/ready`、setup 状态、未初始化时 setup 初始化、已开启的注册/登录 | 调用 `/v1`、读取用户数据、修改配置或通道 |
| 普通用户 User JWT | 管理自己的资料、API Key、日志和账单 | 查看他人数据、调整额度、管理通道、修改 settings |
| 管理员 User JWT | 管理普通用户、用户额度、通道、全局日志和基础看板 | 管理超级管理员、绕过审计修改关键 settings |
| 超级管理员 User JWT | 管理管理员账号、系统 settings 和高风险操作 | 绕过安全校验、绕过审计、泄露密钥明文 |
| API Key | 仅调用 `/v1/*` 模型接口，并受所属用户、Key 预算、额度和通道策略约束 | 调用 `/v0/*`、修改账号/通道/settings、覆盖下游鉴权密钥 |
| Provider 回调签名 | 支付 provider 回调入账，必须校验签名、金额、订单和幂等 | 依赖用户 JWT、信任客户端提交额度 |

### API Key

API Key 是调用 `/v1/*` 的唯一凭据，不使用 User JWT 调用模型。
API Key 生命周期、轮换、泄露处置、作用域、缓存一致性和高级管理以 `docs/API_KEYS.md` 为准；本文只保留跨模块原则。
策略决策顺序、访问控制、用户分组、通道分组、限流、`routerx.route` 冲突规则和策略快照以 `docs/POLICIES.md` 为准。协议兼容、APIType、入口协议和上游能力等级以 `docs/PROTOCOLS.md` 为准。一次调用的路由、策略、usage、计费、错误和 settings 事实快照以 `docs/SNAPSHOTS.md` 为准。

设计约束：

- API Key 格式以 `sk-` 开头。
- 明文只在创建响应中出现一次。
- 数据库保存 SHA256 哈希，不长期保存明文。
- 禁用、删除、过期、用户禁用或额度不足必须立即影响 `/v1` 调用。
- API Key 可以设置最大消耗额度，也可以设置自身无限制；有限 Key 创建时不扣用户余额，调用时同时检查并扣减用户余额和 Key 预算；无限制只表示 API Key 自身不限制，仍受用户额度约束。

### 通道

通道是 RouterX 对下游模型服务的配置实体。商业级通道设计必须支持简单配置和高级配置同时成立。

| 能力 | 说明 |
|------|------|
| 单上游 | `base_url` + `api_key`，适合小白和单 provider。 |
| 多上游 | `base_urls`、`api_keys`、`upstreams`，适合同类服务多节点。 |
| key 选择 | `round_robin`、`random` 等策略，避免单 key 压力集中。 |
| 模型匹配 | `models` 支持逗号列表和 `*`。 |
| 模型重写 | 客户端模型名可映射到上游真实模型名。 |
| 通道分组 | `channel_group` 支撑套餐、倍率、访问控制和路由偏好。 |
| 优先级与权重 | 先按 `priority`，再按 `weight` 做加权选择。 |
| 扩展配置 | `upstream_options` 保存 Azure api-version、Anthropic version 等 provider 配置。 |

下游密钥不得在响应和日志中泄露。配置了 `ENCRYPTION_KEY` 时，下游 API Key 应加密存储。

通道路由的商业级默认顺序是：

```text
通道候选缓存/预加载 -> 系统硬性过滤 -> 管理员访问策略 -> 用户 routerx.route 偏好 -> priority -> weight -> 模型重写 -> 上游解析
```

其中系统硬性过滤包括用户/API Key 状态、用户余额、Key 预算、通道启用状态、模型匹配、Adapter 可用性、熔断和安全策略。`routerx.route` 只能表达偏好，不能覆盖后台策略。通道内部解析按 `upstreams`、多 key/多 base URL、单 key/单 base URL 的顺序处理；`upstreams` 中的 base URL 和 key 视为绑定关系，不与外层数组交叉组合。管理员修改通道、分组、价格或 settings 后，必须刷新通道候选缓存或递增路由版本，避免集群实例长期使用旧候选集。

### Relay

Relay 是核心模块。RouterX 的长期定位是多协议入口与多上游同等重要：

- 入口协议：OpenAI、Anthropic、Gemini。
- 上游厂商：OpenAI、OpenAI-Compatible、Azure OpenAI、Anthropic、Gemini、xAI、Qwen、DeepSeek、RouterX-Compatible。

入口协议决定客户端看到的请求和响应格式；上游厂商决定 RouterX 如何调用下游。二者必须解耦。

阶段原则：

- P0 建立可靠闭环：API Key 鉴权、通道选择、OpenAI-compatible 非流式基础调用、日志和扣费。
- P1 扩展核心矩阵：流式响应、Anthropic/Gemini 入口、主要上游转换和 provider-specific 参数。
- P2 完善高级接口：Images、Audio、Moderations、Responses 深度兼容、企业路由和多层 RouterX。

`routerx` 私有扩展字段用于表达路由偏好和上游扩展参数，真实厂商调用前必须剥离；RouterX-Compatible 上游可以继续透传，并通过 hop 限制避免循环。

Relay 的关键验收不是“能把请求发出去”，而是每次请求都能回答四个问题：为什么这个用户可以调用、为什么选中这个通道、为什么这样计费、失败时应该由用户修正请求还是由管理员修正配置。

`/v1` 错误必须遵守入口协议外形：OpenAI 客户端看到 OpenAI 兼容错误，Anthropic 客户端看到 Anthropic 兼容错误，Gemini 客户端看到 Gemini 兼容错误。RouterX 内部可以统一错误 code，但不能把管理端统一响应包装返回给模型 SDK。

### 计费

计费系统负责把 usage 转成内部额度消耗，并保证日志、账单和余额一致。

核心规则：

- 额度统一使用 `int64` 基础单位，避免浮点误差。
- 成功调用必须写日志并扣费。
- 余额不足必须在调用下游前拒绝。
- 扣费必须使用数据库事务和条件更新，避免并发透支。
- usage 来源优先级为上游标准 usage、适配器转换、本地估算、最低计费规则。
- 价格表达式和倍率是两个层次：表达式计算基础费用，倍率来自用户分组、通道分组和访问策略。

商业级计费必须形成事实链：

```text
鉴权和额度预检 -> 路由与访问控制 -> 上游 usage -> 价格规则快照 -> 条件扣费事务 -> 日志/账单聚合
```

余额不足、访问控制拒绝和通道不可用都必须在调用下游前失败；下游已调用但 usage 缺失时，必须按配置估算或执行最低计费，并记录 usage 来源。历史日志应保存价格表达式、倍率、访问控制和路由摘要的快照，避免规则变更后无法解释旧账单。

在线支付是可选商业模块。Stripe、易支付等支付能力不能成为自部署基础链路的强依赖。

### 日志、审计与观测

调用日志是审计、账单和排障事实表。默认不保存完整请求体和响应体；开启后必须截断和脱敏。

商业级观测目标：

- HTTP 访问日志包含 method、path、status、latency、ip、request_id。
- 模型调用日志包含 user、token、channel、model、usage、quota、status、error。
- 管理端关键操作进入审计日志。
- 后续暴露 Prometheus `/metrics`，覆盖请求量、错误率、耗时、通道可用性、Redis/DB 错误等指标。

## 9. 分阶段蓝图

### P0：自部署可用闭环

目标：系统从空库到第一次模型调用形成可验证闭环。

范围：

- 初始化、统一登录、JWT/API Key 鉴权。
- 用户、管理员、API Key、通道和日志基础管理。
- OpenAI-compatible 模型列表和 Chat 非流式转发。
- 通道密钥加密、API Key 哈希、额度预检、日志记录和基础扣费。
- `/health`、`/ready` 和基础错误格式。

验收：

- 空库可初始化，管理员可登录。
- 管理员可创建通道，用户可创建 API Key。
- 有效 API Key 可调用 `/v1/models` 和基础非流式 Chat。
- 禁用用户、禁用 API Key、禁用通道后立即生效。
- 成功调用写日志并扣额度，余额不足不调用下游。
- 密钥不出现在响应和日志中。

### P1：核心商业能力

目标：形成多协议、多上游、可运营的中转系统。

范围：

- OpenAI、Anthropic、Gemini 三类入口协议基础非流式和流式调用。
- OpenAI-Compatible、Anthropic、Gemini、xAI、Qwen、DeepSeek、Azure OpenAI 等主要上游。
- SSE 流式转发、客户端断开取消、流式 usage 结算。
- 通道重试、熔断、限流和自动恢复。
- 模型价格表、通道价格覆盖、通道分组、用户分组倍率和访问控制。
- 充值码和支付插件的接口设计。

验收：

- 主流 SDK 能通过 RouterX 完成基础调用。
- 不支持的字段明确报错或记录降级原因，不静默误转。
- 计费规则变更不影响历史账单，日志保存规则快照。
- 多通道选择、重试和熔断行为可解释。

### P2：企业与生产增强

目标：满足企业接入、长期运行和规模化运维。

范围：

- OAuth/OIDC、账号恢复、注销保留账号、多会话和登录审计。
- 管理审计日志、结构化日志、Prometheus 指标和告警。
- API Key 哈希迁移兜底、通道密钥轮换、KMS 扩展和生产就绪检查。
- 支付回调幂等、退款记录、人工修正和支付审计。
- 高级 OpenAI API、Realtime、Files、Fine-tuning、多区域路由等候选能力。

验收：

- 关键管理操作可审计。
- 生产配置缺失时 readiness 能准确反映不可用状态。
- 企业身份接入不允许账号接管。
- 支付入账幂等，重复回调不会重复增加额度。

## 10. 商业级验收原则

商业级验收不只看“接口能不能返回 200”，而要看默认体验是否可靠、复杂场景是否可解释、失败时是否安全。

| 维度 | 验收原则 |
|------|----------|
| 开箱体验 | 新用户能按一条主路径完成首次调用，不被高级配置阻塞。 |
| 安全默认 | 密钥不泄露，请求/响应体默认不记录，余额不足不调用下游。 |
| 可解释性 | 通道选择、计费规则、错误来源、限流和熔断状态能被日志或审计解释。 |
| 可恢复性 | 配置错误、上游失败、Redis 缺失/故障、迁移失败都有明确处理策略。 |
| 可扩展性 | 新 provider、新协议、新计费模板通过清晰接口扩展，不破坏已有入口。 |
| 可维护性 | 文档、路线图和代码事实一致，阶段边界不误导实现者。 |
